# 🤖 AIモデル使い分けガイド

> **このドキュメントは何？**
> 開発で使えるAIモデル（Gemini・Claude）には、それぞれ「得意なこと」と「苦手なこと」があります。
> 正しく使い分けることで、開発スピードと品質が大きく向上します。
> このガイドを見て、「今やりたい作業にはどのモデルを使えばいいか」を判断してください。

---

## 🔤 まず知っておきたい用語

| 用語 | 意味 |
|------|------|
| **AIモデル** | 質問に答えたりコードを書いてくれるAIの「頭脳」のこと。モデルごとに性格が違います |
| **Gemini** | Googleが開発したAIモデル。画像や動画も理解でき、Android開発に特に強い |
| **Claude** | Anthropic社が開発したAIモデル。慎重で正確な分析やレビューが得意 |
| **マルチモーダル** | テキストだけでなく、画像・動画・音声なども理解できる能力のこと |
| **コードレビュー** | 書いたプログラムを別の視点でチェックして、バグや改善点を見つけること |
| **リファクタリング** | プログラムの動きは変えずに、コードを読みやすく・きれいに書き直すこと |

## 🎯 タスク別おすすめモデル早見表

### 開発の流れに沿った使い分け

| フェーズ | 具体的な作業 | おすすめモデル | 理由 |
|---------|-------------|--------------|------|
| **① 企画・設計** | アプリの機能一覧を整理する | **Claude** | 要件の抜け漏れを慎重にチェックしてくれる |
| | 画面のレイアウトを考える | **Gemini** | 手描きスケッチの写真を見せて「こんな画面を作って」と指示できる |
| | データベース設計 | **Claude** | 複雑なデータの関係性を整理するのが得意 |
| **② 実装（コードを書く）** | Android（Kotlin）のコードを書く | **Gemini** | Google製なのでAndroid/Kotlinの知識が特に豊富 |
| | iOS（Swift）のコードを書く | **どちらでもOK** | 両モデルとも十分な能力がある |
| | UI（画面の見た目）を作る | **Gemini** | 「この画像みたいなデザインにして」と画像で指示できる |
| | サーバー側の処理を書く | **どちらでもOK** | 両モデルとも十分な能力がある |
| **③ テスト** | テストコードを書く | **Claude** | 細かいエッジケース（例外的な状況）まで考慮してくれる |
| | バグの原因を調べる | **どちらでもOK** | Geminiは素早く、Claudeは深く分析。状況に応じて選ぶ |
| **④ レビュー** | コードの品質チェック | **Claude** | 慎重にコードを読み、問題点を漏れなく指摘してくれる |
| | セキュリティチェック | **Claude** | セキュリティへの意識が高く、危険なコードを見逃しにくい |
| **⑤ ドキュメント** | READMEやマニュアルの作成 | **Claude** | 分かりやすく丁寧な文書を作成するのが得意 |
| | APIドキュメントの作成 | **どちらでもOK** | 両モデルとも十分に対応できる |
| **⑥ リリース準備** | ストア申請文の作成 | **Claude** | 審査基準を考慮した慎重な文面を作ってくれる |
| | プライバシーポリシーの作成 | **Claude** | 法的な文書の作成に慎重で正確 |

## 💡 実践的な使い方のコツ

### Gemini を呼び出すとき
```
（Antigravityのモデル選択で Gemini を選んでから）

「この手描きのスケッチ（写真）を元に、ログイン画面を作って」
「Androidアプリのナビゲーションを実装して」
「Firebase（Google提供のサーバー機能）との連携コードを書いて」
```

### Claude を呼び出すとき
```
（Antigravityのモデル選択で Claude を選んでから）

「このコードにセキュリティ上の問題がないかチェックして」
「アプリ全体の設計を整理して、ドキュメントにまとめて」
「プライバシーポリシーの草案を作って」
```

## ⚠️ 大事な注意点

1. **「正解」は1つではない** — 上の表はあくまで目安です。迷ったら今使っているモデルでまず試してみてください
2. **AIの回答は必ず確認する** — どちらのモデルも間違えることがあります。特に法的文書やセキュリティに関わる部分は、専門家にも確認しましょう
3. **同じ質問を両方に聞いてみるのもアリ** — 重要な設計判断の時は、両方のモデルに聞いて比較すると良い結果が得られます
