# ⚠️ アプリリリースのリスクと対策ガイド

> **このドキュメントは何？**
> アプリを作って Apple App Store や Google Play に公開（リリース）する際に、
> 実際に起こりうる問題と、その対策をまとめたガイドです。
> 「知らなかったせいで公開できない！」とならないよう、開発の早い段階から目を通してください。

---

## 🔤 まず知っておきたい用語

| 用語 | 意味 |
|------|------|
| **リリース** | アプリをストア（App Store / Google Play）に公開して、誰でもダウンロードできるようにすること |
| **審査（しんさ）** | AppleやGoogleがアプリの内容をチェックすること。これに通らないと公開できない |
| **リジェクト** | 審査に落ちること。理由は通知されるので、修正して再提出すれば再挑戦できる |
| **プライバシーポリシー** | 「このアプリはユーザーの情報をこう扱います」という約束事を書いた文書。法律で義務付けられている |
| **利用規約** | ユーザーがアプリを使う上でのルールを定めた文書 |
| **APIキー** | 外部サービス（地図、AI、決済など）を使うための「パスワード」のようなもの |
| **GDPR** | ヨーロッパの個人情報保護法。海外ユーザーがいる場合は対応が必要 |
| **SSL/TLS** | インターネット通信を暗号化して安全にする技術。「https://」の仕組み |

---

## 📋 リスク一覧と対策

### 1. 🚫 審査でリジェクト（不合格になる）リスク

**起こること:** せっかく完成したアプリが、AppleやGoogleの審査に通らず公開できない。

| よくあるリジェクト理由 | 対策 |
|----------------------|------|
| アプリの説明と実際の機能が違う | ストアの説明文は正確に書く。スクリーンショットも実際の画面を使う |
| バグやクラッシュがある | リリース前に十分なテストを行う。色々な端末でテストする |
| プライバシーポリシーがない | 開発の早い段階で作成する（Claudeに草案を作ってもらえる） |
| 不適切なコンテンツがある | Apple/Googleのガイドラインを事前に読んでおく |
| 他のアプリのコピーと見なされた | オリジナリティのある機能やデザインを心がける |
| ログイン機能があるのにテストアカウントが審査時に提供されていない | 審査用のテストアカウントを用意して提出時に記載する |

> [!TIP]
> **対策のコツ:** 最新の審査ガイドラインを Antigravity に「Apple/Google の最新の審査基準を調べて教えて」と聞いてみましょう。

---

### 2. 🔒 セキュリティ（安全性）のリスク

**起こること:** ユーザーの個人情報が漏れたり、不正アクセスされたりする。

| リスク | 対策 |
|--------|------|
| APIキーがソースコードに直接書かれている | APIキーは環境変数（`.env` ファイル）に保存し、`.gitignore` に登録して GitHub に上がらないようにする |
| 通信が暗号化されていない | 必ず HTTPS（SSL/TLS）を使う。HTTP は使わない |
| ユーザーのパスワードがそのまま保存されている | パスワードは「ハッシュ化」（元に戻せない形に変換）して保存する |
| SQLインジェクション（データベースへの攻撃） | ユーザー入力を直接SQLに使わず、必ず「パラメータバインディング」を使う |

> [!CAUTION]
> **絶対にやってはいけないこと:** APIキーやパスワードなどの秘密情報を GitHub にアップロードすること。
> 一度アップロードすると、削除しても履歴に残ってしまいます。
> `.gitignore` ファイルの設定を最初にしっかり行いましょう。

**`.gitignore` に入れるべきもの例:**
```
# 秘密情報
.env
*.keystore
google-services.json
GoogleService-Info.plist

# ビルドファイル（容量が大きいため）
/build/
/node_modules/
```

### 3. 📜 法的リスク（法律に関するリスク）

**起こること:** 法律違反でアプリが公開停止になったり、罰金が発生する。

| リスク | 対策 |
|--------|------|
| プライバシーポリシーがない | **必須。** どんな小さなアプリでも作成する。ユーザーの情報を一切取らなくても必要 |
| 他人の著作物（画像・音楽・フォントなど）を無断使用 | 必ずライセンスを確認する。フリー素材でも「商用利用可」か確認する |
| 個人情報保護法への未対応 | 日本の個人情報保護法に準拠する。海外展開するならGDPRにも対応する |
| 利用規約がない | サービスの条件を明確にするため作成する |
| オープンソースライセンス違反 | 使っているライブラリのライセンスを確認し、必要に応じてアプリ内に表記する |

> [!IMPORTANT]
> プライバシーポリシーと利用規約は **アプリ公開に絶対必要** です。
> Antigravity（Claudeモデル）に「このアプリ向けのプライバシーポリシーの草案を作って」と頼むと下書きを作ってくれますが、
> 最終的には **法律の専門家（弁護士など）に確認してもらうことを強く推奨** します。

### 4. 💰 費用に関するリスク

**起こること:** 予想外の費用がかかり、開発や運用が継続できなくなる。

| 費用項目 | 内容 | 目安 |
|---------|------|------|
| Apple Developer Program | iOSアプリ公開に必須の年間費用 | **年額 ¥15,800**（2025年時点） |
| Google Play Console | Androidアプリ公開に必須の登録費用 | **$25（約¥3,800）一度きり** |
| サーバー費用 | Firebase や AWS などクラウドサーバーの利用料 | 無料枠あり。ユーザーが増えると課金発生 |
| 外部API費用 | AI、地図、決済などの外部サービス利用料 | サービスにより異なる |
| デザイン素材 | 有料アイコン、フォント、画像など | ものによる |

> [!WARNING]
> **サーバー費用の意外な落とし穴:** Firebase などは無料枠がありますが、アプリが人気になってアクセスが急増すると、
> 突然高額な請求が来ることがあります。必ず **利用上限（Budget Alert）を設定** しておきましょう。

---

### 5. 📱 技術的リスク

**起こること:** 特定の端末で動かない、遅い、クラッシュするなどのトラブル。

| リスク | 対策 |
|--------|------|
| 古いバージョンのOSで動かない | サポートするOSの最低バージョンを決めて、それ以上でテストする |
| 特定の画面サイズで表示が崩れる | 複数サイズの端末でテストする（小さいスマホ〜大きいタブレット） |
| アプリが重い・遅い | 画像の最適化、不要な処理の削減を行う |
| オフラインで全く使えない | ネットワーク未接続時の動作を事前に設計する |

---

### 6. 🔄 運用リスク（公開後に起こるリスク）

**起こること:** 公開後にバグが見つかったり、サーバーが止まるなどのトラブル。

| リスク | 対策 |
|--------|------|
| 公開後にバグが見つかる | アプリにクラッシュレポート機能を入れておく（Firebase Crashlytics が便利） |
| サーバーが止まる | サーバーの稼働監視を設定する。障害時の連絡方法を決めておく |
| ユーザーからの問い合わせ | メールアドレスやフォームなどの問い合わせ窓口を用意する |
| アップデートの計画がない | 最低でも月1回はバグ修正アップデートを行う計画を立てる |

## ✅ リリース前チェックリスト

リリースする前に、以下を全て確認しましょう。

- [ ] プライバシーポリシーを作成し、アプリ内からアクセスできるようにした
- [ ] 利用規約を作成した
- [ ] APIキーなどの秘密情報が GitHub に上がっていないか確認した
- [ ] `.gitignore` を正しく設定した
- [ ] 複数の端末・OSバージョンでテストした
- [ ] クラッシュレポート機能を組み込んだ
- [ ] ストアの審査ガイドラインを読んだ
- [ ] ストア用のスクリーンショットと説明文を用意した
- [ ] 問い合わせ先（メールアドレスなど）を用意した
- [ ] サーバー費用の利用上限（Budget Alert）を設定した
- [ ] 使用しているライブラリのライセンスを確認した
- [ ] テストアカウントを審査用に用意した（ログイン機能がある場合）
